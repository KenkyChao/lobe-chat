# 企业版中的 Codex

Codex 是当前企业版中已经明确可用的编程助理能力之一。它沿用官网 Codex 文档的核心工作方式，但在企业版分支中，会和桌面端品牌、执行设备、工作区与系统工具入口一起出现。

本文只描述已经能够确认的能力；仍在持续调整的部分统一放在文末的**待确认项**中。

## 已确认功能

### 1. 运行方式

- Codex 通过**桌面端**运行
- NaiYun AI 会在本地以子进程方式启动 `codex` CLI
- 运行结果会回流到聊天界面，而不是停留在外部终端

### 2. 使用前提

- 需要安装 **Codex CLI**
- 需要确保 `codex` 命令已经加入 `PATH`
- 需要先在终端中至少运行一次 `codex` 完成登录
- 如果本机没有检测到 Codex CLI，系统会弹出安装引导

安装命令沿用官网基线：

```bash
npm install -g @openai/codex
```

或在 macOS 上使用：

```bash
brew install --cask codex
```

### 3. 创建入口

- 首页会出现带有**编程助理**属性的推荐入口
- 也可以从**创建助理**流程中手动创建 Codex 助理
- Codex 助理与普通问答助理不同，更适合仓库修改、命令执行与测试验证

### 4. 工作目录

- 每个 Codex 会话都绑定一个**工作目录**
- 该目录会被视为项目根目录
- 发出第一条消息前，建议先选好工作目录
- 如果在对话过程中切换工作目录，当前话题会自动开启新的 Codex 会话

### 5. 聊天中的可视化反馈

企业版分支已明确支持将 Codex 的关键执行结果渲染成专用区块：

- **文件变更**：显示新增、删除、修改、重命名，以及每个文件的增减行数
- **待办列表**：显示已完成、进行中、待处理的任务项
- **命令执行**：显示命令、退出码、stdout、stderr
- **子助理**：支持并行子任务，并在对话中独立展示

### 6. 会话恢复

- 同一话题中的 Codex 会话支持跨消息继续
- 你可以在上一轮任务完成后继续追问，例如补测试、改文档、继续重构
- 以下情况会触发自动新会话：
  - 工作目录发生变化
  - 原始线程已不存在
  - CLI 返回 `no conversation found` 或 `thread not found`

### 7. 执行设备

当前界面已明确支持以下执行设备：

- **本机**：在当前桌面端本地运行
- **云端沙箱**：在临时云端环境中运行，企业部署默认可用
- **远程设备**：运行在通过 `lh connect` 接入的其他机器上

<Callout type={'info'}>
  执行设备入口已经存在，但具体能否选择到某个设备，取决于当前部署能力和设备在线状态。
</Callout>

## 推荐使用流程

<Steps>
  ### 第一步：先准备 CLI

  在本机安装 Codex CLI，并在终端中完成一次登录。只有终端里的 `codex` 可用后，桌面端里的 Codex 才能正常启动。

  ### 第二步：创建 Codex 助理

  从首页推荐入口或创建助理流程中添加 Codex。建议一个代码仓库对应一个 Codex 助理，便于长期恢复上下文。

  ### 第三步：确认执行设备与工作目录

  先决定这次任务要在本机、云端沙箱还是远程设备上运行，再选定具体工作目录。

  ### 第四步：发送明确任务

  适合使用完整指令，例如 “修复这个报错并补测试” 或 “把这段逻辑抽成可复用 hook，并更新相关文档”。

  ### 第五步：通过反馈区块跟进进度

  优先关注文件变更、待办卡片和命令输出，它们是判断 Codex 当前做了什么、还剩什么的最快入口。
</Steps>

## 使用建议

- **优先固定仓库与目录**：工作目录一旦频繁变动，会明显影响会话恢复效果
- **尽量一仓库一助理**：比 “一个助理处理所有项目” 更容易维护上下文
- **先确认设备是否可用**：远程设备需要先完成接入；云端沙箱默认可用
- **多看反馈区块**：CLI 缺失、登录失效、设备离线等问题，通常都会先体现在系统提示或执行结果里

## 限制说明

- **仅桌面端可用**：Web 端无法启动本地 Codex CLI
- **登录态与 CLI 共享**：终端里的 Codex 失效，桌面端里的 Codex 也会受影响
- **会话绑定工作目录**：不会自动跨目录或跨机器跟随
- **设备能力受接入状态影响**：远程设备是否可用，取决于目标机器是否已经完成接入

## 待确认项

- 企业版是否需要单独强调 Codex 与 Claude Code 的职责边界
- 工作目录切换时的提示文案是否还会继续调整
- 安装指引中的 “打开系统工具 / 重新检测” 交互是否已经定稿
